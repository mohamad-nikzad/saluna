# Android Contact Picker for Client Import (PWA)

Managers can add salon **Clients** from their phone address book on Android via the [Contact Picker API](https://developer.mozilla.org/en-US/docs/Web/API/Contact_Picker_API), in addition to the existing VCF file bulk import. **iOS is out of scope** — Safari does not expose `navigator.contacts`; we rely on feature detection, not user-agent checks, and show the picker only when `navigator.contacts?.select` exists. Bulk import uses a phone-first entry on `/clients/import` but keeps VCF as fallback; single pick adds «از مخاطبین گوشی» to `ClientPicker` (search mode) and `ClientDrawer` (create only). Both paths map **Device Contact** data to name + canonical phone only (same as VCF), reuse `buildClientImportPreview` and the bulk API for multi-select, and handle duplicates explicitly: auto-select an existing Client in the booking picker, confirm before linking in the client drawer. Canceling the OS picker is a silent no-op. The installed PWA requires `"permissions": ["contacts"]` in `manifest.webmanifest`; we rejected iOS workarounds, CSV import, and pulling email or other fields into Client records because they expand scope without matching the existing import contract.

**Considered options:** VCF-only (status quo — forces export steps on Android); native Expo app contacts module (native app is not in prod); UA-gated or standalone-only picker (rejected — the API availability check is sufficient and works in Android Chrome tabs too).

**Consequences:** New `salon-core` mapper from picker results to `VcfDraftContact`; PWA owns picker invocation and single-pick duplicate UX; no API or schema changes.
