# AppointmentRequest is a distinct aggregate, not an Appointment status

Public, customer-submitted bookings are stored in their own `appointment_requests` table with their own status enum, not as a new status on `appointments`. The alternative — adding a `pending_public` status to `appointments` — would have forced every existing read model (calendar, today, conflict checks, revenue, retention) to learn to exclude that status, turning every untouched query into a latent bug. A request also has invariants that an appointment doesn't: no validated `clientId` (just raw phone), no assigned staff at submit, and it must never block availability or appear on the staff calendar. Keeping it a separate aggregate makes those invariants enforceable at the type and table level, and lets the existing `Appointment Intake` pipeline stay the single validation gate by running again at approval time (which may legitimately 409 if the world has moved on).
